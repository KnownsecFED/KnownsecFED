服务端渲染和静态化

NodeJSsuntopo2018-03-25

这里不说 React 的 SSR，说下在 hapi 框架下使用 hapi-react-views 静态化网站时的一个小 tip。

hapi-react-views 的 render

服务端渲染其实很简单，就是把数据和模板结合的一个过程，最简单的例子可以用 ES6 的字符串模板来理解。

hapi-react-views 的基本用法文档写的很详细，不赘述，这里着重看下 render 的方法。根据文档有一下 4 种渲染的方式:

  1. reply.render(‘index’, data)
  2. request.render(‘index’, data)
  3. server.render(‘index’, data)
  4. manager.render(‘index’, data)

无论那种方式，首先要调用 render 方法，然后将模板和数据传入到 render 方法中。但是 hapi 自身并没有 render 方法，而是 hapi-react-views 本身提供的，然后注入到 hapi 中并挂载到 hapi 不同的对象上，来方便我们获取到 render 方法而已。

实际上渲染这个过程与 hapi 并没有太多的直接关系，只要能够拿到 render 方法，我们就能够得到一系列的 html 字符串，然后可以通过 hapi 的 reply 返回给前端。抛开框架，简单示意下流程:

const hapiReactViews = require('hapi-react-views');
...
async (request, reply) => {
  const render = hapiReactViews.render;
  const htmlString = render('index', data);
  return reply(htmlString);
}

注: hapi 在注册 views 时，注意 server.root.views 和 server.views 的区别，否则会报错。

如何静态化

网站的静态化就是把整个网站所有可访问的 URL 返回的 HTML 都已文本的方式保存。在细粒度上控制可以有全站静态化，模块静态化，详情静态化，定时静态化，按时间段静态化等。

实现方式简单来说有两种方式:

  1. 遍历访问所有 URL
  2. 遍历渲染所有的 render 方法

第一种方式缺点显而易见，每个 URL 都要走完整的 HTTP 请求，耗时较长，也给服务器带来不必要的压力，但是好处是代码耦合度低。 第二种方式，直接调用 render 保存数据，优点是速度快，缺点是耦合度高。

实践

hapi 下实现服务端渲染和静态的化的关键在于如何获取 render 方法。虽然有四种方式都可以获取 render 方法，但是并不是每一种都能实现不同细粒度上的静态化。举个例子: 定时静态化 + reply.render(或 request.render) 由于定时静态化并没有 http 请求，所以此时是不存在 request 和 reply 对象，更不用说 render 方法了。那么四种 render 方式，只剩下两种可以选择，这里选择第三种，在 server 对象上获取 render 方法。

第一步，定义公共的 service 层方法，专门渲染输出静态 html。该设计将渲染工作和具体模板引擎独立开，无论使用或更换什么渲染引擎，只需要传入对应渲染引擎的 render 即可。

/**
 * 传入相应的渲染引擎 render，以及对应的业务参数
 * 返回 html 字符串
 * renderService.js
 */
export const renderIndex = async options => {
  const {render, params} = options;
  // 业务层代码 -- 根据 params 参数获取 render 数据
  const data = getRenderData(params);
  return render('index', data);
}

第二步，定义 controller 方法，实现 SSR。

/**
 * 第一种 controller 层的实现方式
 * 不使用 service 渲染层，而是直接在 controller 中渲染
 * 优点：简单直接，缺点：无法公用 service 层方法，功能分层不明显，抽象度低
 * indexController.js
 */
export const indexController1 = async (request, h) => {
  const {params} = request;
   // 业务层代码 -- 根据 params 参数获取 render 数据
  const data = getRenderData(params);
  return h.render('index', data);
}

/**
 * 第二种，调用公共 service 层方法，推荐这种方式
 * service 层抽象了公共的方法，所有需要 html 的地方都可以调用，代码简洁，分层明显
 * indexController.js
 */
const requestWrapper = request => {
  // 强行将 render 的执行上下文绑定为 request
  const render = request.render.bind(request);
  request.render = render;
  return request;
};

export const indexController2 = async (request, h) => {
  const {params} = request;
  // 在 controller 中，从 request 中获取 render 方法，并强行绑定 render 方法的执行上下文
  const {render} = requestWrapper(request);
  // 调用 service 渲染层方法
  const content = await renderService.renderIndex({render, params});
  return reply(content);
}

第三步，定义静态化 schedule 层并以插件的形式插入到 hapi 中。server 中 render 可以类似理解为全局的属性，不需要具体的 request 和 reply.

const register = async (server, options, next) => {
  // 通过 server 获取 render 方法，并强行绑定 render 的执行上下文
  const render = server.render.bind(server);
  // 调用 service 渲染层，由于公用方法，减少了很多代码量
  const content = await renderService.renderIndex({render, params: {} });
  fs.writeFileSync(indexPath, content);
  return next();
}
register.attributes = {
  name: 'schedule',
};

export default register;

总结

在同时需要处理服务端渲染和静态化的时候，有几个需要注意的知识点：

  1. hapi 注册 views 时，注意 server.root.views 和 server.views 用法，到底有啥子区别？
  2. javascript 中函数执行上下文绑定，如 server.render.bind(server)，不这样绑定会怎样？
  3. service 渲染层的抽象，渲染通常不是在 controller 层吗？
